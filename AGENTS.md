# AGENT.md

## 项目概述

- 理念：一个简单但不普通、精致且量身定做的记账软件，力求用户最小心智负担，获得最佳体验。（项目名为 moni，意为 money-mini）。
- 技术栈（多采用最新稳定版）：采用原生开发，使用 Rust 后端（重后端内核）与Kotlin（仅用于编写安卓原生UI外壳以及平台特殊代码）。以 Rust 为跨平台业务逻辑与全局状态机，Kotlin 仅作为无状态 UI 渲染层与原生系统能力的‘桥接代理’。
- 代码设计：代码库高度模块化解耦，可维护性强。
- UI风格：安卓使用原生 Material 3 Expressive。
- 平台：现阶段专注安卓开发；同时rust内核需整理接口，保留逻辑跨平台潜力，但现阶段只做潜力培养和保持，不写其他平台实质性代码。

1. 以达到用户的目标为准，不要怕麻烦而选择效果差的方案；不要过度谨慎，过多采用v1最小化设计等。
2. 尽可能不要引入新的警告、严禁引入错误。
3. 确保代码最佳实践、可读性、可维护性、健壮性、性能、安全性等质量要求。
4. 由于你没有实时获取信息的能力，你的训练数据可能包含过时的信息，例如新的稳定版已经发布、技术栈已更新、方法失效等等问题，若遇到困难无法确定，请善用搜索，或询问用户，以用户的反馈为准。
5. 输出语言为中文；注意选择 `UTF-8` 或 `UTF-8 with BOM`，避免乱码。
6. 如果在开发过程中有需要长时间或未来需要引用的文档或重要内容，可以在 `.docs\tmp` 创建一个'YYYY-MM-DD_HH-mm-`<slug>`.md'文档记录，以便引用和回看。
7. 与项目相关的文档统一放入项目根目录的 `docs` 文件夹，积极维护，文档分目录整理，并设置 README 索引。
8. 每次开发完毕后编译一个 debug 签名的 arm64 release apk 到 `/home/agguy/share/moni-release`，然后提交所有更改。

## 核心开发指令与规范

> **Role & Rule**: 作为资深架构师，输出代码必须绝对遵循以下规范，无例外。

### 1. 黄金法则
- **高内聚低耦合**：功能内聚，依赖抽象。
- **DRY 原则**：拒绝重复代码，通用逻辑必须抽取。
- **单一职责**：一个类/函数只做一件事。
- **防御性编程**：不信任外部输入，核心边界必须有严格校验。

### 2. 架构与解耦
- **领域边界**：按业务领域划分目录，绝对禁止模块间循环依赖。
- **最小权限**：类与方法默认 `private`，仅通过明确定义的接口对外暴露。
- **数据隔离**：严禁将数据库实体直出前端，必须经 DTO 转换。
- **依赖注入**：面向接口编程，严禁在业务逻辑中硬编码实例化复杂服务对象。
- **配置分离**：环境参数必须外部化，严禁硬编码魔法值。

### 3. 代码健壮性
- **控制规模**：函数 < 50 行，类 < 300 行。
- **拒绝嵌套**：使用卫语句（Guard Clauses）尽早返回，杜绝 `if-else` 嵌套超 3 层。
- **异常与日志**：严禁静默吞没异常，关键分支与异常捕获处必须附带上下文日志。
- **异步统一**：统一使用 `async/await`，禁止混用 Promise 链式调用。

### 4. 测试驱动与隔离
- **自动化优先**：勤于编写自动化测试，最大程度减少对人工测试的依赖。
- **物理隔离**：测试文件严禁与源码平铺，必须放入所属模块独立的 `tests/` 子目录。
- **映射一致**：测试文件与源码文件命名严格对应。
- **逢变必测**：核心逻辑的增改必须同步覆盖测试用例，严禁提交“零测试”业务代码。

### 5. 语义与可读性
- **命名即文档**：变量名描述内容，函数名描述动作。严禁无意义单字母、缩写或拼音。
- **有效注释**：注释只解释“为什么 (Why)”，不解释“是什么 (What)”。
- **契约说明**：Public 接口必须包含标准文档注释（参数、返回值、异常）。
- **待办标记**：统一格式为 `// TODO: [具体描述]`。

### 6. 文档沉淀与演进
- **分级定位**：严格区分边界。**短期文档**（Issue、PR）定位为“沟通备忘”，重在说明业务上下文，拒绝过度设计与撰写；**长期文档**（架构图、数据字典、核心契约）作为核心资产，需保持严谨，指导项目演进。
- **适度与有效**：拒绝长篇大论与形式主义。践行“代码说明 How，文档说明 Why”，只记录最核心的业务逻辑与“设计权衡”。
- **积极维护**：文档与代码同频共振。核心代码变更必须强制同步更新关联文档；过期内容或废弃方案立即清理并归档，坚决消灭误导人的“假文档”。
- **沉淀提炼**：建立转化机制。短期文档依托极简模板融入日常工作流；当核心模块或大版本迭代稳定后，再定期将短期的核心决策，提炼沉淀至长期文档中。
- **就近原则**：推行文档即代码（Docs-as-Code）。文档尽可能随源码库托管，保持单一事实来源，降低团队的查找与上下文切换成本。
